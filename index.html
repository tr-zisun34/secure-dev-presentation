SQUARE Methodology: Nine Steps

SQUARE (Security Quality Requirements Engineering): A systematic nine-step process for defining and eliciting comprehensive security requirements.

üî¢ Complete Nine-Step Process

1. Agree on Definitions

Establish common terminology from sources like RFC 4949 to prevent miscommunication among team members with different backgrounds.

2. Identify Assets & Security Goals

Define what needs protection and associated security objectives for different stakeholders (e.g., HR focuses on personnel data confidentiality).

3. Develop Artifacts

Create documentation including concept of operations, project goals, usage scenarios, threat scenarios, and misuse cases.

4. Perform Risk Assessment

Select appropriate risk assessment methods using artifacts from step 3. Threat modeling supports this process to identify high-priority exposures.

5. Select Elicitation Technique

Choose appropriate method based on stakeholder diversity (structured interviews for cultural differences, ARM for most successful results).

6. Elicit Security Requirements

Apply the selected requirements elicitation technique following detailed guidance provided by the chosen method.

7. Categorize Requirements

Distinguish between essential requirements, goals (desired requirements), and architectural constraints to support prioritization.

8. Prioritize Requirements

Use cost/benefit analysis and methods like Analytical Hierarchical Process (AHP) with pairwise comparisons for effective prioritization.

9. Requirements Inspection

Perform formal inspection (Fagan inspections most effective) to produce final set of prioritized security requirements.

E-commerce Example - Complete 9 Steps:

  • Step 1: Define "customer data" to include payment info, addresses, purchase history
  • Step 2: Identify customer database as critical asset with confidentiality goals
  • Step 3: Create data flow diagrams showing payment processing flow
  • Step 4: Assess SQL injection risks in payment forms using OWASP methodology
  • Step 5: Use structured interviews with diverse stakeholder teams (IT, business, security)
  • Step 6: Conduct interviews to gather requirements like "Payment data must be encrypted in transit and at rest"
  • Step 7: Categorize as essential (PCI compliance) vs. desired (fraud detection)
  • Step 8: Prioritize encryption requirements over advanced analytics based on regulatory compliance costs
  • Step 9: Perform Fagan inspection to validate final security requirements list

üè¢ Compartmentalization Example: OpenSSH

Architecture Strategy: OpenSSH divides system functionality into distinct intercommunicating subsystems:

  • Privilege Separation: Different components run with minimal required privileges
  • Process Isolation: Critical operations isolated in separate processes
  • Attack Surface Reduction: Each component has limited exposure to external threats
  • Blast Radius Containment: If one component is compromised, damage is contained

Saltzer's 8 Security Principles

Foundational guidelines from Saltzer and Schroeder for secure system design

üîí Least Privilege

Give programs/users only the minimum privileges necessary to perform their specific job functions

‚öôÔ∏è Economy of Mechanism

Keep security designs as simple and small as possible - complexity breeds vulnerabilities

‚ùå Fail-Safe Defaults

Default access settings should deny access - systems should "fail closed" rather than "fail open"

üóùÔ∏è Separation of Privilege

Require multiple keys/conditions to unlock critical functions - no single point of failure

üîç Complete Mediation

Check every access attempt to every resource, every time - no exceptions or shortcuts

üìñ Open Design

Security mechanisms shouldn't depend on secrecy of design - "security through obscurity" is weak

üîó Least Common Mechanism

Minimize shared components that could become single points of failure across security boundaries

üë§ Psychological Acceptability

Security mechanisms must be user-friendly - if too complex, users will find workarounds

Real-World Application: A banking system implements least privilege by giving tellers access only to customer account viewing, not administrative functions, while requiring dual approval (separation of privilege) for large transactions.

  • Architecture: Web frontend ‚Üí API gateway ‚Üí Microservices ‚Üí Database cluster
  • Decomposition: Trust boundaries between internet/DMZ, DMZ/internal network, application/database

  • Rate & Prioritize:
    • T1 Priority: CRITICAL - High impact (payment data breach) + High likelihood (common attack vector) = Risk Score: 9/10
    • T2 Priority: HIGH - Medium impact (account takeover) + Medium likelihood (requires network access) = Risk Score: 6/10
